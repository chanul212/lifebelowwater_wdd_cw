2. Feedback & Programme Directory

Technical Description

The page is divided into three main sections: the Programme Directory, the Ratings section, and the Feedback Form. The Programme Directory uses a scrollable container (.programme-scroll) with max-height: 520px; overflow-y: auto and custom scrollbar styling via ::webkit-scrollbar to keep the list compact without losing content. Each programme card uses a CSS Grid layout with three columns (thumbnail image, text content, action button), collapsing to a single column on screens below 700px via a media query.

The feedback form uses two <fieldset> elements to logically group personal details and feedback content fields, with <legend> labels for both. A live character counter is implemented in JavaScript using an input event listener on the textarea, subtracting the current length from 500 and updating a <p> element in real time. The counter text turns red when fewer than 50 characters remain.

Form validation is fully JavaScript-driven. On submit, the handler calls e.preventDefault(), then validates the name field (non-empty), email field (against a regex pattern /^[^\s@]+@[^\s@]+\.[^\s@]+$/), and message field (non-empty). Invalid fields receive a red border via inline style and show an adjacent error paragraph by setting style.display = 'block'. On successful validation, a confirmation message is revealed and the form is reset.

Accessibility

All form inputs have associated <label> elements linked via for/id attributes. The programme card list uses role="list" and each card uses role="listitem" for screen-reader landmark compatibility. The confirmation message uses role="alert" so assistive technologies announce it automatically when it appears. The sort dropdown uses aria-label for screen-reader identification.

3. Team Page

Technical Description

The team page uses a CSS Grid with repeat(auto-fit, minmax(220px, 1fr)) to lay out four team member blocks responsively. Each .member-block is position: relative and contains both the circular portrait thumbnail and a hidden .member-details overlay panel.

The hover reveal effect is achieved entirely with CSS transitions. The .member-details panel starts with opacity: 0; max-height: 0; overflow: hidden and transitions to opacity: 1; max-height: 400px when the parent .member-block receives the :hover or :focus-within pseudo-class. Simultaneously, the thumbnail transitions to opacity: 0.4; transform: scale(1.06) to give a blur-out effect that draws attention to the overlay. No JavaScript is required for this interaction.

The overlay panel uses position: absolute anchored to the top of the member block, and transform: translateX(-50%) to centre it horizontally regardless of the block's position within the grid. pointer-events: none is set by default and switched to auto on hover, preventing the invisible panel from intercepting mouse events when not shown.

Accessibility

Each member block has tabindex="0" so keyboard users can focus it and trigger the :focus-within state to reveal the details panel. The overlay panel uses role="tooltip". Each block includes an aria-label indicating how to access details (hover or focus). This ensures the interactive content is usable without a pointing device.

4. Content Page - Plastic Pollution & Ocean Waste

Technical Description

The content page follows the same two-column CSS Grid layout as the other content pages, with a 240px sticky side navigation and a 1fr article column. The side nav uses position: sticky; top: 100px; height: fit-content, so it remains visible without scrolling past the article's natural height.

Section headings use <h2> with a left teal border (border-left: 4px solid var(--teal) and padding-left: 14px) and the seafoam colour variable for contrast. Subsections use <h3> in cyan. Stat callout boxes (.stat-box) use semi-transparent navy backgrounds with the large .stat-num displayed in seafoam to create immediate visual impact for key statistics. A .go-top fixed-position anchor provides persistent navigation for the long-form content.

Accessibility

The side navigation uses aria-label="Page sections" to distinguish it from the site-wide navigation. Images have detailed alt text. On screens below 768px the side nav is hidden and the layout becomes single-column for readability. All anchor links in the side nav target heading IDs using scroll-behavior: smooth defined on the html element in the shared CSS.

5. Challenges & Lessons Learned

Challenge 1: Form validation preventing scroll to error. After showing validation errors, the user had to manually scroll to find the highlighted field. Resolved by adding element.scrollIntoView({ behavior: 'smooth' }) on the first invalid field found, improving error discoverability.

Challenge 2: Team overlay clipped on small screens. The absolutely-positioned overlay panel overflowed the visible area on mobile. Resolved by adding a max-width media query that reduces the overlay width from 240px to 200px on screens below 600px.

Challenge 3: Scrollable programme list height on mobile. The fixed max-height on the programme scroll container left too little visible content on short mobile screens. Resolved by using a media query to reduce max-height proportionally on screens below 600px.

Lesson: CSS-only interactions (like the team hover panels) are more reliable across devices than JavaScript-managed show/hide, but require careful testing at multiple breakpoints to catch overflow and clipping issues.

6. Accessibility Considerations

All images on my pages include descriptive alt text. Form inputs have properly associated labels. Interactive elements are keyboard accessible — the team page blocks use tabindex and :focus-within, while form buttons are native elements. Semantic HTML landmarks (<nav>, <main>, <footer>, <article>, <section>, <fieldset>) are used throughout. The confirmation message on the feedback form uses role="alert" for screen-reader announcement.
